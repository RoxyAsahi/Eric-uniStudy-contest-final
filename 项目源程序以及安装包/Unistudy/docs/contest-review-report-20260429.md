## 2. 项目概况

项目名称：UniStudy。  
形态：Electron 桌面端个人 AI 学习终端。  
主要技术：JavaScript、Node.js、Electron、原生 HTML/CSS/DOM、libSQL、本地文件处理、OpenAI-compatible 模型服务、embedding/rerank 检索增强。  
核心学习链路：Source 资料导入 -> Chat 对话学习 -> 高表现力渲染 -> Notes 笔记沉淀 -> 测验/闪卡/学习日志/记忆。

仓库规模：

- `src` 中 JS/HTML/CSS 约 142 个文件、约 81,019 行。
- `tests` 中 JS 测试约 66 个文件、约 18,801 行。
- Git 跟踪文件约 469 个。
- 当前工作树已有多处未提交修改，评审按当前工作树状态进行。

## 3. 符合项与优势

### 3.1 问题价值与主题契合

项目聚焦多学科复习中的真实问题：资料分散、问答零散、笔记难沉淀、复习难连续。作品定位不是普通聊天工具，而是把资料、对话、笔记、练习和学习记忆放进一个连续流程，主题契合“人工智能技术与实际场景结合”的要求。

加分点：

- 学习场景真实，目标用户和使用过程清楚。
- “话题级 Source 绑定”能减少不同资料混杂，提高学习问答的上下文相关性。
- 桌面端形态适合长时间学习、文件导入和本地资料管理。

扣分点：

- 目前缺少真实用户试用记录、效果数据或学习效率对比。
- “多学科复习”的问题较大，当前材料还可以进一步明确一个最强演示场景，例如数学错题复盘或英语阅读资料复习。

### 3.2 AI 技术应用与功能实现

项目中的 AI 使用不是停留在“调用聊天模型”，而是覆盖了：

- OpenAI-compatible 多 Provider 配置。
- 聊天、标题生成、后续追问、笔记分析、测验生成、闪卡生成等任务模型。
- PDF/DOCX/文本资料解析、切片、embedding、向量检索、可选 rerank。
- Source 检索结果注入聊天上下文，并保留引用信息。
- DailyNote 工具协议、学习日志、日记投影与学习记忆检索。

加分点：

- AI 能力在实际学习流程中发挥作用，功能链路比较完整。
- 支持模型服务抽象，不强绑定单一厂商。
- 知识库检索链路有明确实现：解析、chunk、embedding、相似度排序、阈值、rerank、引用。
- 主进程、preload、renderer 分层已经形成，自动化测试覆盖明显多于一般青少年项目。

扣分点：

- 核心 AI 效果缺少定量评估，例如检索命中率、引用准确性、题目生成质量、错题复习效果。
- Source 检索依赖外部 embedding/rerank 服务；如果现场网络或密钥不可用，核心能力会明显退化。
- 当前存在硬编码 API Key，需立即撤销并改为环境变量或用户本地设置，否则不适合公开提交。

### 3.3 创新性与设计水平

项目创新主要体现在产品设计和学习流程，而不是原创算法：

- 话题级资料绑定。
- 聊天结果直接沉淀到 Notes。
- Notes 再派生分析、选择题和闪卡。
- 高表现力 AI 输出，支持 Markdown、HTML 片段、Mermaid、KaTeX、Three.js 等学习呈现。
- 多 Agent、多 Topic 的连续学习组织方式。

评价：创新性较强，尤其适合青少年创新实践作品；但与 NotebookLM、通用 AI 学习助手、RAG 知识库产品相比，底层技术路线并非全新。参赛材料应避免把“集成多个成熟能力”包装成原创 AI 算法，而应强调“面向复习闭环的系统设计创新”。

## 5. 安全与规范性问题

### 5.1 硬编码密钥

源码中发现测试/预设服务 API Key 被硬编码在至少以下位置：

- `src/modules/main/utils/modelService.js`
- `src/modules/renderer/app/settings/settingsController.js`
- `src/modules/renderer/app/store/appStore.js`

风险：密钥可能被公开传播、滥用或计费；也会削弱作品原创与合规说明的可信度。  
建议：立即撤销该密钥，改为本地环境变量、用户设置导入或演示专用临时凭据；提交材料不得包含真实可用密钥。

### 5.2 Renderer 暴露面较宽

`src/preloads/shared/roles.js` 中 `LITE_KEYS` 暴露的能力超过 80 个，包含文件、剪贴板、窗口、Source、Notes、学习日志、watcher 等。虽然已有 `contextIsolation: true`、`nodeIntegration: false`、`sandbox: true`，但 preload 面仍然偏宽。

建议：按 `chatAPI / sourceAPI / notesAPI / settingsAPI / windowAPI` 分域，敏感能力加参数 schema 和主进程白名单。

### 5.3 任意路径读取风险

`src/modules/main/ipc/fileDialogHandlers.js` 中 `get-text-content`、`get-file-as-base64` 等路径仍接收 renderer 传入的文件路径，缺少严格根目录白名单或主进程签发 token。

风险：一旦 renderer 内容注入被利用，可能读取本地文件。  
建议：改为“文件 ID / attachment token -> 主进程内部解析路径”，不要让 renderer 传真实路径。

### 5.4 HTML 预览安全边界仍需收紧

HTML 代码块预览通过 iframe `srcdoc` 执行；Three.js 预览显式使用 sandbox，但普通 HTML 预览未看到同等 sandbox 配置。项目的核心卖点是“AI 生成可交互学习界面”，因此这里必须特别谨慎。

建议：普通 HTML 预览也设置 sandbox；禁止不必要的 `allow-same-origin`；所有 `postMessage` 做来源、类型、frameId 白名单校验；将“学习互动组件”与“任意 HTML 代码预览”分级处理。

### 5.5 外链协议过宽

`open-external-link` 允许 `http:`、`https:`、`file:`、`magnet:`。对学习应用而言，`file:` 和 `magnet:` 不应由 renderer 普通内容直接触发。

建议：默认仅允许 `https:`；`file:`、`http:`、自定义协议需用户确认或主进程白名单。

## 7. 分项评分理由

### 问题价值与主题契合：17/20

真实学习痛点明确，学习闭环设计完整，AI 与学习场景结合紧密。扣分主要来自缺少用户验证数据和演示场景聚焦不足。

### AI 技术应用与功能实现：19/25

RAG、模型服务抽象、笔记生成、测验/闪卡、学习日志与记忆均有实装，工程深度超过一般作品。扣分来自外部服务依赖、AI 效果缺少指标、测试红灯以及硬编码密钥。

### 创新性与设计水平：16/20

以学习流程为核心的系统整合有创新价值，高表现力渲染有辨识度。扣分来自底层算法创新有限，以及部分体验仍更像工程探索而不是稳定产品。

### 作品完整性与规范性：13/20

源程序和文档较完整，测试体系较强；但当前自动化未全通过，正式参赛材料缺项，工作树未固化，依赖和安全问题仍需修复。

## 8. 建议优先修复清单

1. 修复 `renderer-diary-wall-controller` 失败测试，确保日记墙 Agent 筛选恢复。
2. 修复或解释受控 E2E 的刷新快捷键失败，至少让 `npm test` 全绿。
3. 移除并撤销硬编码 API Key，改为本地设置或环境变量。
4. 处理 `@xmldom/xmldom` high 漏洞，确认 `mammoth` 与打包链路可升级。
5. 收紧文件读取 IPC：renderer 不应直接传任意本地路径。
6. 给 HTML 预览 iframe 加安全 sandbox 与消息白名单。
7. 准备正式提交材料：诚信承诺书、环境说明、AI 使用说明、团队分工、演示视频。
8. 建立可复现参赛版本：清理工作树、打 tag、记录 Node/npm/Electron 版本和模型服务配置。

## 9. 评委结论

UniStudy 是一个完成度和工程量都较高的 AI 学习工具项目，选题符合人工智能创新实践项目要求，且能展示较强的软件工程能力。它最有竞争力的部分不是“用了大模型”，而是把 AI 问答、资料检索、笔记沉淀和复习训练组织成一个持续学习流程。

但以严格评审标准看，当前版本还不能作为最终提交版。主要原因是质量门禁未通过、硬编码密钥和安全边界问题会影响作品规范性，正式提交材料也尚未齐全。建议在修复上述问题后再提交终版；若这些问题得到解决，项目有潜力获得较高评价。
